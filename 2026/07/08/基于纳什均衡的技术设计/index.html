基于纳什均衡的技术设计

分享者: 李广庆

一、架构设计的核心哲学

系统演化的必然性

好架构的标准：支持升级演变＞全知全能

  • 好的架构不是能解决所有问题

  • 应该支持升级与演变, 因为变化才是必然的

  • 好的系统都是具有优美的特点的

纳什均衡与经济学系统
  • 软件服务其实是一个经济学系统

软件即经济：

将代码、服务和资源视为一个动态演化的数字生态系统，用经济学思维（稀缺性、激励、交换）指导设计：

  1. 资源即商品，分配策略决定系统效率；

  2. 服务即企业，API契约是交易规则；

  3. 调度器即政府，通过政策维护系统稳定。
    这种视角让开发者更关注系统的动态平衡，而非静态结构，从而构建出更具韧性和适应性的软件。

系统的演变成本与收益如何达到一个平衡点?

  • 开发者博弈模型

稳定点理论：数据层＞接口层＞服务层＞技术基建

稳定点方法论设计系统: 区分变化与不变

ⅰ. 接口层应该是业务服务的稳定点(包括进出的接口层)
ⅱ. 数据是业务逻辑服务的稳定点, 能用数据去解决的,就不要用计算去解决,数据(接口)定义了问题
ⅲ. 业务与技术(框架,DB,语言), 业务其实才是稳定点, 业务被视为是最重要的, 是核心, 最稳定的, db, 框架等都是可变的, 可以随时根据项目的情况替换掉

  • 纳什均衡启示：多因素动态平衡中的最佳稳定态

用稳定点方法论理解SOLID原则

单一职责原则(SRP)的稳定点分析

稳定机制：

  • 每个类只有一个改变理由 → 减少变化传播路径

  • 修改影响局部化 → 偏离成本高于收益

  • 系统自然收敛到模块化状态

开闭原则(OCP)的博弈平衡

稳定点特性：

  • 核心模块封闭 → 成为系统不动点

  • 扩展接口开放 → 允许策略变化而不破坏均衡

  • 违反惩罚：修改核心导致系统稳定性破坏

里氏替换原则(LSP)的策略一致性

  • 当你扩展一个类时， 记住你应该要能在不修改客户端代码的情况下将子类的对象作为父类对象进行传递

稳定机制：

  • 子类必须遵守基类约定 → 形成策略不变点

  • 破坏契约的子类会被自然淘汰（被客户端拒绝）

  • 系统收敛到所有派生类型可互换的状态

接口隔离原则(ISP)的收益优化

稳定点分析：

  • 最小接口依赖 → 减少变化传播

  • 客户端只承担必要契约 → 无动机偏离

  • 胖接口导致”免费搭车”问题 → 破坏系统稳定性

依赖倒置原则(DIP)的收敛过程

稳定机制：

  • 抽象接口成为系统不动点

  • 高层和低层策略在抽象层达到最优响应

  • 违反DIP导致双向依赖 → 系统进入振荡状态

23条设计模式都可以用稳定点原则去理解

案例1: 装饰模式：装饰者模式、装饰器模式、Wrapper、Decorator

装饰是一种结构型设计模式,允许你通过将对象放入包含,行为的特殊封装对象中来为, 原对象绑定新的行为

案例2: 策略模式

策略是一种行为设计模式，它能让你定义一系列算法，并将每种算法分别放入独立的类中，以使算法的对象能够相互替换。

假如你需要前往机场。你可以选择乘坐公共汽车、预约出租车或骑自行车。这些就是你的出行策略。你可以根据预算或时间等因素来选择其中一种策略

策略与装饰的区别: 策略是可以很多种随时替换的, 装饰是一次性选择

当一个需求来的时候怎么设计?

只有一条原则: 找出需求的稳定点(区分变化与不变的东西)

识别变化热点：将频繁变化的区域隔离为独立策略

稳定点设计法则

数据驱动原则：用存储代替计算（例：业务规则用配置表实现）

抽象固化原则：面向接口编程＞具体实现（Go语言集合设计范式）

核心稳定三角：业务领域模型＞API定义>技术基建(db,框架,语言)
